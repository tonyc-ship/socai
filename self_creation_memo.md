Self Creation

### 背景：
StageHand, Browser Use/Harness, Agent Browser都是通用web use agent，AI可以自定义操作任何网页。
随着Claude、GPT模型越来越强，并且在持续优化CUA能力，很多通用操作任务都可以基本完成。
但缺点是思考过程慢；复调用多模态理解能力，token成本高；遇到复杂的操作流程和复杂网页形态，操作失败率就上去了。
所以基本上还停留在一次性demo层面，对于真正每天高频进行自动化网页操作的严肃工作，可用性很低。

socai的思路是更实际的：
首先搭建通用CDP底座。然后对每个站点，沉淀高效、高稳定性的操作流程，做到每次操作非常快，并且token成本为0或很低。
目前先实现了小红书站点。

### 目标：
让每一个socai的用户，都能实现添加一个新网站、或新的操作流程，用户不需要懂任何技术。
过程需要AI的参与，可以是用户自己的Claude Code、Codex等vibe coding助手，也可以用socai内置的agent。

### 当前方案：
socai提供了一套Harness，包括三部分：
1. Snapshots。包含DOM、截图（分为用户视窗截图和完整元素截图）、a11y tree三部分。页面状态每次变化、或每一个单步操作，都记录一次。
2. Code Scaffold。包括文件结构和职责分类，有registry, entities, tools, page_scripts, knowledge等。另外有完整的xhs站点代码作为参考。
3. Development Cookbook。在creation/SKILL.md详细描述了代码开发流程，具体是一个开发-验证-观察的3-step loop，避免LLM在其他web use工具里常出现的猜测、heuristic、乱试等行为。

### 验证结果：
我使用GPT 5.5进行了验证，初始Prompt大致是 “请执行core/src/sites/creation/SKILL.md 这个skill，帮我新增支持抖音”
一共进行了5版验证，每版根据遇到的问题，我不断优化上面的方案。每版结果分别push到了代码分支dy1, ..., dy5。
最后一版的结果还不错，one shot基本上完成了任务。
但是需要考虑用户中还有使用其他略差的模型，结果更不可控。即使GPT 5.5，在多次测试中，也每次存在不一致的对问题的理解和代码结果。

### 存在的问题：
上面的抖音 self creation虽然初步验证成功，但是过程中能观察到很多struggles，并且由于和小红书流程比较相似，所以抄了作业。如果在不同类型的网站上，可能出现更多问题。
观察到的问题：
- 大型网站的DOM非常复杂，一个页面的全部DOM可达1MB+文件大小，精简后也还有可能100KB+，对应约30K+ LLM tokens，
- 页面形态多变，需要兼容没见过的unkown unknowns。比如抖音、小红书的搜索页的布局会动态变化，当输入不同的搜索词条，除了原本的帖子瀑布流，有的词条会额外动态展示AI总结、用户搜索、单列或多列布局等等
- 流程细节太多，页面如何稳定操作，需要沉淀成知识，比如是点击键盘按键，还是点击页面按钮；有的UI元素需要鼠标悬浮到某区域才出现；点错了如何回到上一步等等
- 页面风控增加了不确定性，比如抖音主页会打不开，转圈圈若干分钟才打开；小红书显示“这里是一篇荒地”或者“你要找的内容不见了”。
- 人类和Agent/LLM一个关键区别是，人类在浏览网页时会进行大量的实时探索和微操。比如人类在打开一个陌生网页时，会微微往下翻、鼠标来回滑动，这里那里点一点，很快就知道各个元素是如何布局的了。但是agent缺少和页面的动态交互来探索页面，对LLM来说，每一步都很重，每执行一个小动作，都需要复杂的多模态理解截图或理解超长DOM，要花费分钟级别，而对人类只是一秒内的事；或者agent习惯于执行一个完整操作后再理解页面，但是中间很多瞬态UIUX变化都漏掉了，导致无法找到要点击的位置，或者找不到出错原因。
